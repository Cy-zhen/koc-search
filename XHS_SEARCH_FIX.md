# 小红书搜索返回 0 结果：问题诊断与解决方案

## 1. 现象描述

在本地运行（`localhost:3000`）并且平台显示“已登录”状态下，精准搜索例如“潮玩”等关键词，进度条跑到 100% 后，提示“**触发平台验证/风控，请先在弹出的页面手动完成验证后重试**”，并且最终返回 **0** 个结果。

## 2. 根因诊断

通过 SSE 接口追踪日志和自动浏览器（Playwright）现场截图，发现了核心阻断点：

1. **强制登录弹窗 (Login Overlay)**
   小红书近期在网页版搜索结果页（`https://www.xiaohongshu.com/search_result...`）加入了一个强势的 DOM 层弹窗：“**登录后查看搜索结果**”。
   即使本地应用通过检测 Cookie (`web_session`) 认为你是“已登录”状态，这个弹窗依然可能因为风控、Cookie 权限不足或会话过期等原因弹出来。

2. **DOM 遮挡导致元素无法抓取**
   由于 Playwright 脚本（`xiaohongshu.js`）只是在后台默默“等待列表加载”并“滚动”，它完全没有预料到会被一个弹窗挡住。脚本在弹窗下方尝试寻找对应的 CSS 选择器（`.user-list-item` 等），自然什么也找不到（抓取到 0 个）。

3. **代码误判为“风控”**
   连续滚动 8 次后未拿到任何数据，代码进入 `diagnoseZeroResult()` 阶段。由于该方法里正则表达式命中了弹窗文本中的“扫码登录/手机号登录/请先登录”，它把这个情况错误地归档到了“触发平台风控”或者系统错误，从而导致任务无状态退出。

4. **真实数据其实存在**
   人工介入验证发现，只要手动叉掉那个弹窗（或者点击弹窗周围），其实**背景里的搜索结果是完全加载出来的**（比如能看到几万粉的潮玩博主），只是代码没有去“揭开”这个盖子。

## 3. 解决方案思路

既然数据在弹窗下面，我们的核心思路就是：**像真人一样把弹窗关掉**，然后再开始抓取。

具体在代码上的落地分为三个层面：

### 层面一：实现“弹窗终结者”（`dismissLoginOverlay`）
在 `xiaohongshu.js` 的搜索主流程（进入对应搜索链接之后，寻找数据之前）插入一个拦截方法。采取循序渐进的四种策略消除弹窗：

- **策略 1（点按钮）**：遍历常见的弹窗关闭选择器（如 `.close-button`, `[aria-label="关闭"]`, `svg` 图标，甚至是“暂不登录”按钮）。如果发现可见，立刻点击并等待。
- **策略 2（敲键盘）**：按 `Escape` 键。现代 UI 弹窗大多支持 ESC 关闭。
- **策略 3（点遮罩）**：模拟鼠标点击屏幕边缘（比如 `x:10, y:10`），触发点在外部遮罩层的情况以关闭弹窗。
- **策略 4（核弹级：直接删 DOM）**：如果前面都没用，直接用 JS 在浏览器上下文中把 `body` 里看起来像覆盖层的超大尺寸 `div` 给 `remove()` 掉，并解除 `body` 的 `overflow: hidden` 限制。

### 层面二：增强 CSS 选择器兼容性
防止小红书页面结构更新导致的老选择器失效：
- **Tab 选项卡**：加入了泛用的 `.tab`, `a`, `li` 等带有“用户”文本的定位方案。
- **用户卡片**：增加了对 `section[class*="user"] a`, `[class*="search-result"]` 等全新结构的备用选择器，确保只要有超链接指向 `/user/profile/` 就能抓出来。

### 层面三：提升反检测水位（Stealth）
为了进一步降低被弹窗针对的概率，在 `browser-manager.js` 中增加伪装：
- 伪装 `navigator.connection`（网络类型状态）。
- 伪装 `navigator.maxTouchPoints`（触摸点）。
- 抹除框架默认附带的特有全局变量（如 `__playwright` 等）。
- 补充常见的 Chrome `loadTimes` 和 `csi` 等插件特征。

## 4. 结论与现状

经过以上的靶向改造：
当检索遇到小红书强制登录框时，程序能自动处理并移除拦截，随后正常完成页面的滚动与数据的刮削，顺利完成搜索闭环。
